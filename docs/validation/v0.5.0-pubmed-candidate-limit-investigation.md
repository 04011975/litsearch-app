# PubMed Candidate Limit Investigation

## Test setup

Query: cancer  
Source: All Sources  
Sort: Relevance

## Results

| Candidate limit | Raw | Deduped | Duplicates removed | Warm total_ms |
|---------------|-----:|---------:|-------------------:|--------------:|
| 2000 | 2400 | 2336 | 64 | ~2992 ms |
| 1000 | 1400 | 1337 | 63 | ~4447 ms |
| 500 | 900 | 837 | 63 | ~4373 ms |

## Observations

- Lower candidate limits significantly reduce the number of available deduplicated records.
- End-to-end latency does not improve meaningfully.
- Warm-run latency is dominated by OpenAlex and Semantic Scholar.
- PubMed is no longer the primary bottleneck after retrieval parallelization.

## Conclusion

No change recommended.

Keep candidate limit at 2000.

Reducing the PubMed candidate limit significantly reduces available
deduplicated records while providing no meaningful end-to-end latency
improvement because OpenAlex and Semantic Scholar dominate warm-run latency.
